## 2. user-profile v1.0.0 ‚Äî per-invariant notes

- **B-PRO-001 (compose-only)** ‚Äî `composeProfile({ user, mine, stored, now })`
  serves exactly: `name/addedAt` from the user-account record (read-only; the
  account bundle is untouched and frozen), `pbs` = `computePbTable(mine)` and
  `aggregates` = `computeAggregates(mine)` ‚Äî the EXACT result-stats handshake
  payloads (pass-through objects `{pbs:[...]}` / `{modes:[...]}`), `xp` =
  `round2(Œ£ xpOf(r))` over the user's stored results (the sealed B-LB-007
  per-result field), `streaks` from `computeActivity(mine).days` ONLY.
  Nothing derived is persisted: a new result is reflected on the next read
  (tested: xp grows immediately after a new submission; deep-equality of the
  pass-throughs against `/api/stats/*` responses asserted).
- **B-PRO-002 (streaks)** ‚Äî implemented exactly as sealed: active day =
  `testsCompleted >= 1` on that UTC date in the activity series; current =
  consecutive active days ending at the most recent active day, alive iff that
  day is today or yesterday UTC (injected clock), else 0; max = longest run in
  the series. Single source of truth (round-7 ruling): the only input is the
  result-stats activity series ‚Äî there is no independent computation path.
  Module tests cover gaps, duplicates/unordered input, 0-count days, and the
  UTC-midnight boundary (¬±1ms); HTTP tests boot with a pinned clock, advance
  it across windows, and cross-check against `/api/stats/activity`.
- **B-PRO-003 (level)** ‚Äî documented delegated curve: `level = floor(sqrt(xp /
  10))` with `XP_PER_LEVEL_SQ = 10` (level n requires `xp >= 10¬∑n¬≤`; level 1 at
  10 xp, 2 at 40, 10 at 1000). Monotonic non-decreasing by construction
  (swept 0‚Üí200k xp in tests), deterministic, integer ‚â• 0, recomputed per read,
  never stored. Coefficients isolated to one constant ‚Äî a reference-grounded
  curve lands as a one-line amendment (B-QT-007 discipline).
- **B-PRO-004 / S-PRO-003 (strict edits)** ‚Äî `validateProfileUpdate` mirrors
  `profile-update.schema.json` closed: keys ‚äÜ {bio, avatarUrl, socials,
  isPublic}, minProperties 1, bio ‚â§500, avatarUrl ‚â§500 ‚àß `^https://` (URL
  reference only ‚Äî no blob storage), socials ‚äÜ {website(^https://), twitter,
  github} ‚â§200, isPublic boolean. Any violation ‚áí 422 ErrorEnvelope with ZERO
  fields written (all-or-nothing tested per bad-case: profile byte-identical
  after); exactly one store write on success (`max_writes_per_request: 1`);
  unauthenticated ‚áí 401 before validation; edits target only the requester's
  own record (uid-keyed).
- **B-PRO-005 (privacy)** ‚Äî `GET /api/profile/:username` is unauthenticated
  and serves only the sealed public shape (ajv-checked; a leak sweep asserts
  no uid/pw/email/token/moderator/isPublic/hash material anywhere in the
  payload). Unknown name OR `isPublic=false` ‚áí the IDENTICAL 404 ErrorEnvelope
  (same status/code/message; only the random correlation_id differs ‚Äî
  O-RES-004 precedent), tested both directions; the owner's authenticated read
  is unaffected; flipping the toggle back restores visibility. Lookup is
  case-insensitive, mirroring account lookup (delegated, ¬ß4.1).
- **S-PRO-001/002, O-PRO-001/002** ‚Äî profile + every failure ajv-conform to the
  sealed schemas in tests; no outbound network; writes confined to
  `data/profile.json`; composition is bounded in-process work (well inside
  p95 ‚â§ 100ms).
- Routes (delegated surface, ¬ß4.1): `GET /api/profile` (own),
  `PATCH /api/profile` (edit), `GET /api/profile/:username` (public).

## 3. public-api v1.0.0 ‚Äî per-invariant notes

- **B-API-001 (lifecycle)** ‚Äî `POST /api/apekeys` (session-gated) mints
  `pdd_` + 32 hex chars (128-bit entropy, ruled BQ-API-01) and returns
  `{ key, apekey }` ‚Äî the plaintext appears in that response ONLY (list and
  every later surface carry metadata only; asserted by string sweep). At rest:
  per-key 128-bit salt (hex) + `sha256(salt + "." + plaintext)`; the test
  reads `data/apekeys.json` off disk and asserts the plaintext is absent while
  `hashKey(stored.salt, key) === stored.hash`. 1000-mint uniqueness sweep.
  `GET /api/apekeys` lists own metadata (ajv-conform to apekey.schema.json ‚Äî
  never salt/hash/uid). `DELETE /api/apekeys/:id` revokes (`enabled=false`),
  idempotent (second revoke ‚áí 200), unknown/foreign id ‚áí indistinguishable
  404; revoked or unknown keys never authenticate (identical 401 ‚Äî tested).
- **B-API-002 (authentication)** ‚Äî `authenticateApeKey` recomputes the salted
  hash for EVERY stored record and compares equal-length sha256 digests with
  `timingSafeEqual`, never short-circuiting: duration depends only on the key
  count, not on prefix-match length or match position. The sealed statistical
  timing gate (tolerance band) is the validator stage's; the candidate's cheap
  proxy test asserts the structural property ‚Äî median auth time for a match on
  record #1 and #200 of 200 is within 3√ó of a no-match full scan (an
  early-exit implementation would show >50√ó). Domain separation: session
  tokens on `/api/public/*` ‚áí 401 (not key material); ApeKeys on
  session-gated routes fail `verifyToken` ‚áí 401 (tested over results/config/
  stats/profile/apekeys endpoints).
- **B-API-003 (scopes, fail-closed)** ‚Äî closed vocabulary {results,stats,
  profile,quotes}:read. Out-of-scope ‚áí 403 ErrorEnvelope; every scope opens
  exactly its own surface (4√ó5 matrix tested); unknown scope names are
  rejected at creation (S-API-001). A key with zero matching scopes accesses
  nothing (creation requires ‚â•1 valid scope; scope check precedes the
  per-key rate consumption).
- **B-API-004 (parity)** ‚Äî the mirrors call the SAME derivation closures as
  the source routes (refactor ¬ß1), so parity is structural. HTTP tests assert
  byte-equality of `/api/public/{results,results/pbs,stats/*,profile,quotes,
  quotes/random?seed=}` against the session surfaces over a fixture carrying
  normal + minThresholdFailed + bailed + zen submissions ‚Äî exclusion semantics
  ride along (zen absent on both surfaces, flags persisted/visible, PB
  exclusions identical, approved-only quotes identical, tag filters mirrored).
- **B-API-005 / O-API-003 (rate limits)** ‚Äî documented delegated constants:
  fixed 60s windows; **60 req/window per key**, **120 req/window per source
  IP** (IP ‚â• key ‚Äî the per-key dimension stays the tested contract). Window
  accounting is `floor(now/windowMs)` ‚Äî deterministic under the injected
  clock (module tests hit exact window edges ¬±1ms, reset, dimension
  independence, stale-window eviction). Over-limit ‚áí 429 ErrorEnvelope
  (`rate_limited`, ajv-conform ‚Äî the envelope is closed, so retry metadata
  rides standard headers: `Retry-After` + `X-RateLimit-{Limit,Remaining,
  Reset}`, asserted consistent with the window). HTTP tests: 61st request on
  one key ‚áí 429 (second key unaffected; window flip re-enables); two keys at
  60+60 from one IP trip the IP ceiling on request 121 (a fresh key is still
  429 ‚Äî the IP dimension is checked first and does not consume key quota; the
  same fresh key from a different IP ‚áí 200; unauthenticated requests from the
  hot IP ‚áí 429). Check order (documented): per-IP ‚Üí key auth (401) ‚Üí per-key
  (429) ‚Üí scope (403).
- **S-API-001/002/003** ‚Äî create requests mirror the sealed create schema
  (unknown scopes/keys/types ‚áí 422; nothing persisted on failure; duplicate
  scopes are schema-legal and stored verbatim); every API-surface failure
  (401/403/429/404) ajv-conforms to the sealed envelope; mirrored payloads are
  the source handshake shapes by construction (and ajv-checked at source in
  the wave-2 suite).
- **O-API-001/002** ‚Äî no outbound network; store writes confined to
  `data/apekeys.json` (one per lifecycle mutation; ZERO on the read surface ‚Äî
  rate counters are in-memory per process/isolate, documented; `lastUsedAt`
  deliberately omitted, ¬ß4.9); read surface adds bounded auth+counter work
  over the source budgets (well inside p95 ‚â§ 150ms).
- Routes (delegated surface, ¬ß4.5): management `POST/GET /api/apekeys`,
  `DELETE /api/apekeys/:id`; mirrors `GET /api/public/{results, results/pbs,
  stats/{aggregates,pbs,activity,wpm-series}, profile, quotes, quotes/random}`
  (Bearer ApeKey).

## 4. Delegated decisions (CA-001 ‚Äî decided + logged)

1. **Profile route names** (unsealed surface): `GET/PATCH /api/profile`,
   `GET /api/profile/:username`; public-name lookup **case-insensitive**
   (mirrors user-account's case-insensitive name handling).
2. **Level curve**: `floor(sqrt(xp/10))` ‚Äî simplest documented monotonic
   curve with quadratic level costs; coefficient isolated as
   `XP_PER_LEVEL_SQ` for a one-line reference-grounded amendment.
3. **xp inclusion rule**: summed over ALL the user's stored results
   (bailed/flagged included) ‚Äî the sealed sentence reads "over the user's
   stored results" without exclusions; mirrors B-STS-002's sealed inclusion
   default for activity-style aggregates (xp is a lifetime-activity measure).
4. **PATCH semantics**: provided top-level keys replace stored ones
   (defaults-merge read); `socials` is replaced wholesale when present.
5. **Public-API surface prefix** `/api/public/*` + Bearer presentation:
   cleanly separated auth domain (fail-closed guard trivially auditable) with
   zero collision against the session surface; management under
   `/api/apekeys` (session-gated, per the bundle's reference-posture note).
   Creation response shape `{ key, apekey }` (201; show-once), revoke returns
   the metadata with `enabled:false`.
6. **Rate constants**: 60/key/60s + 120/IP/60s fixed windows. Retry metadata
   in `Retry-After` + `X-RateLimit-*` headers because the sealed ErrorEnvelope
   is closed (`additionalProperties:false`) ‚Äî the O-RES-004 envelope precedent
   forbids payload extensions.
7. **Check order**: per-IP window ‚Üí key auth ‚Üí per-key window ‚Üí scope.
   Rationale: the IP dimension guards the whole surface incl. auth probing;
   scope failures must not consume the per-key tested budget.
8. **Rate counters in-memory** (per Node process / per Workers isolate):
   O-API-001 confines *store* writes to the ApeKey store and counters ‚Äî
   in-memory counters are not store writes; the only writable capability file
   is `data/apekeys.json`, and per-request read-surface writes would violate
   the read posture. Documented limitation: counters do not survive restarts
   and are per-isolate (windows are 60s; the per-key contract is enforced
   wherever the key is used).
9. **`lastUsedAt` omitted** (schema-optional): tracking it would put a store
   write on every authenticated read (write amplification against the read
   posture for display-only metadata). Additive if a later minor wants it.
10. **Client-IP source**: Node honors `x-forwarded-for` (first hop) else the
    socket address; the worker prefers `cf-connecting-ip`. Documented; the
    per-IP dimension is an abuse floor, not an identity claim.
11. **Salt/hash form**: 128-bit per-key salt (hex), `sha256(salt + "." +
    plaintext)` ‚Äî the sealed "salted hash" without prescribing a KDF; sha256
    suffices for 128-bit-entropy keys (no dictionary surface; HMAC/PBKDF adds
    no property the invariant seals). Constant-time comparison is over the
    32-byte digests.
